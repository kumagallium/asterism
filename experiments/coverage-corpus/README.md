# coverage-corpus — is the Tier 0 function library "enough"?

Track C of the Tier 0 expansion (see `handoff_to_claude_code_tier0_functions.md`).
The declarative substrate routes "real computation" through a **closed, vetted**
function library (`asterism.functions`, Tier 0). This experiment turns the
question *"is that small head of functions enough for arbitrary datasets?"* from
an opinion into a **measured gate** over a diverse, cross-domain corpus.

It depends only on the project's **stable** interfaces (`inspect`, `materialize`,
`rml_check.load_registry_fn_iris`) — never on the internals of `functions.py` or
the `propose` system prompt. So it is decoupled from Track A (core functions) and
Track B (primitives): those sessions edit `functions.py` / the prompt; this one
only *reads the outputs*, and its numbers improve automatically as A/B land.

## Layout

```
coverage-corpus/
  datasets/<id>/source/<file>     # the downsampled public data (see SOURCES.md)
  datasets/<id>/domain.md         # the propose §2 domain hint for that dataset
  fetch_corpus.py                 # reproducible fetch + deterministic downsample
  runs/<id>/inspection.md         # deterministic inspect output (asterism-inspect)
  runs/<id>/prompt.md             # the exact propose user message (system + user)
  runs/<id>/proposal.md           # the model's schema proposal (its §9 RML is read)
  report/coverage.md , coverage.json   # the aggregate report
```

The analyzer + CLI live in the step0 package: `asterism_step0.coverage`
(`asterism-coverage`), unit-tested in `step0/tests/test_coverage.py`.

## How it works (LLM generation is decoupled from analysis)

1. **prepare** (deterministic, free) — `asterism-coverage prepare` runs
   `asterism-inspect` on each dataset and writes `runs/<id>/inspection.md` plus
   `runs/<id>/prompt.md` (the verbatim `propose` system + user message).
2. **propose** (the LLM step) — author `runs/<id>/proposal.md` by following
   `prompt.md`. This corpus's proposals were generated by the **Claude Code
   subscription agent acting as `propose.SYSTEM_PROMPT`** (no Anthropic API
   billing). The `AnthropicLLMClient` path stays available for automated runs.
   Proposals are committed so the report is reproducible without re-running any
   model.
3. **report** (deterministic, free) — `asterism-coverage report` extracts each
   proposal's §9 RML (via `materialize`), parses it (rdflib), and aggregates the
   metrics below. Re-runnable any time, no key needed.

```bash
cd step0
uv run asterism-coverage prepare --corpus ../experiments/coverage-corpus \
                                 --runs   ../experiments/coverage-corpus/runs
# … author runs/<id>/proposal.md for each dataset …
uv pip install -e ../ingest   # so the T9 closed-set is the live REGISTRY
uv run asterism-coverage report --corpus ../experiments/coverage-corpus \
                                --runs   ../experiments/coverage-corpus/runs
```

## What is measured

From each proposal's RML mapping, deterministically:

1. **`…Raw` fallback rate** — of the columns the proposal treated as needing
   real computation (a vetted function **or** a `…Raw` fallback), the fraction
   that fell back to the raw literal. This is the headline number the gate is
   defined on. `computed = function_maps + raw_fallbacks`; `raw_rate = raw / computed`.
2. **T9 misses** — function IRIs the proposal *referenced* but that are **not**
   in the closed set (`rml_check.load_registry_fn_iris`). The prompt forbids
   inventing functions, so a compliant model rarely produces these; when it
   does, it is both a guard-rail check and a Tier 1→Tier 0 promotion signal.
3. **Per-function usage** — how often each vetted function is used across the
   corpus (which head functions earn their place vs. which are starrydata-only).

Plus a conservative, **secondary** *demand sniffer* (`sniff_demand`) over the
inspection that categorises columns by the transform they appear to need
(messy_date, epoch_millis, boolean, doi, url, value_with_unit, multivalue/json,
…) and cross-checks how the proposal handled each. **It never feeds the gate** —
see the caveat below for why it is needed anyway.

## The "enough" gate

> **Of the columns a proposal treats as needing real computation, fewer than
> `RAW_RATE_GATE` may fall back to a `…Raw` literal.**

**Initial value: `RAW_RATE_GATE = 0.15`** (`coverage.DEFAULT_RAW_RATE_GATE`).

Rationale:

- It is an **A/B acceptance target, not the current state.** The current Tier 0
  (8 starrydata-shaped functions) scores **~64%** on this corpus — see the
  snapshot below — so the gate is comfortably FAILING today, by design.
- 15% (the loose end of the handoff's suggested 10–15%) tolerates the genuinely
  *irreducible* fallbacks — array-of-object cells like Crossref `author` that
  need an RML **nested TriplesMap / multi-value expansion**, not a scalar
  function — while still demanding that the easy wins (single-element arrays,
  comma-wrapped lists, date-parts) be covered by Track B primitives.
- **Recalibrate after each A/B landing.** Re-run `report`; as primitives cover
  the comma-list and single-element-array cases the rate should drop well under
  15%. Once the corpus is larger (more computation-heavy datasets, so the
  denominator is less dominated by 2 files) tighten toward 10%.

Tune it without editing code: `asterism-coverage report --gate 0.10`.

### ⚠️ Caveat — the gate under-counts *scalar* demand

The `…Raw` fallback (and therefore the gate) only fires for the columns the
`propose` prompt tells the model to fall back on: **multi-valued / nested**
cells. A *scalar* column that needs cleaning but has no function (epoch-millis →
dateTime, a bare DOI, a `MALE/FEMALE` boolean, a `300 K` value+unit) is simply
mapped to a **direct literal** — invisible to `raw_rate`. That demand is exactly
what Track A's head functions target, so it surfaces in the **demand-by-category**
table instead (rows handled `direct`/`unmapped` in a category with no covering
function). Read the gate (Track B / expansion signal) and the demand table
(Track A signal) **together**. After Track A adds those functions, the affected
columns move from `direct` → `function`, which *also* lowers `raw_rate` by
growing the computed denominator.

## Snapshot (current Tier 0 = 8 functions, this corpus)

`report/coverage.{md,json}` holds the live numbers. As of the first run:

- **Gate: corpus `…Raw` rate 63.6% ≥ 15% → ❌ FAIL** (7 raw fallbacks of 11
  computed columns). Concentrated in `crossref-works` (100%) and `earthquakes`
  (60%); 8 of 12 datasets have no computation-needing columns at all.
- **Per-function usage:** only `fn:date_iso` (2) and `fn:iri_safe` (2) are used.
  `fn:qudt_quantity` / `fn:qudt_unit` / `fn:float_array_*` / `fn:slug` go
  **unused** — empirical confirmation that the array/QUDT functions are
  starrydata-specific (and that `slug` is largely subsumed by Morph-KGC's
  automatic template encoding).
- **T9 misses:** none — the compliant proposals never referenced an undefined
  function. The closed-set guard-rail holds; demand shows up in the next table.
- **Demand by category (Track A targets):** `epoch_millis` ×2, `value_with_unit_name`
  ×4, `boolean` ×2, `doi` ×1 are all mapped `direct`/`unmapped` (no covering
  function); `multivalue_or_json` ×4 went to `…Raw`. `messy_date` ×2 and `url`
  ×2 are already satisfied by `date_iso` / `iri_safe`.

## Honest limitations

- **Proposals are agent-authored, not API-sampled.** They follow the real
  `propose.SYSTEM_PROMPT` and the real per-dataset inspection, so they are
  representative, but a different model/temperature would vary at the margins.
  Re-generate via the `AnthropicLLMClient` path to cross-check.
- **The corpus is author-curated.** Datasets were chosen to span domains and to
  *include* computation-needing columns; the absolute rate reflects that mix.
  The gate is calibrated on this corpus and should be re-checked if the corpus
  changes materially.
- **The demand sniffer is heuristic** (regex over a few sample values). It is
  deliberately conservative (a space is required for value+unit so ID codes like
  `1000chmg` do not match) and is advisory only.
