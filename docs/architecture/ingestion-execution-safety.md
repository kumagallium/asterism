# 取り込みの実行安全境界 — 生成 ingester を自動実行しない

決定: 2026-06-01 / 設計セッション（UI 担当の調査を受けて core が判断）
status: **合意済み（暫定 = option 1 / 北極星 = option 2 は Phase 5）**

本書は「step0 が生成する ingester をどう安全に実行するか」を確定する。背景は、Phase 4 UI で「設計 → 取り込み → Ask」のループを UI から回す案を検討した際に判明した、**AI 生成コードの実行（RCE 級）リスク**である。

## TL;DR

### 結論

1. **AI が生成した ingester を自動実行しない。** これが鉄則。生成 ingester（実行可能な Python）は、人間がレビューし、人間が明示的に実行する経路のみを通す。
2. **暫定 = option 1（人間承認ゲート）。** UI は「設計 → 生成物 DL → 人間が検証して既存 watcher 経由で取り込み → M2 履歴に出て Ask」を繋ぐだけ。取り込みエンジンは作らない。UI 側だけで安全に実装できる。
3. **北極星 = option 2（宣言的マッピング + 検証済みエンジン、Phase 5）。** データ経路から codegen を消す最終形。ただし n=1（starrydata のみ）で設計すると外すため、2 個目 dataset（Supercon）投入とセットの Phase 5 で core 主導で着手する。
4. **ARiSE デモには option 1/2 のどちらも不要。** デモは投入済みの starrydata を引くだけで、UI の取り込みループは critical path に無い。

### 採用しない案

- **生成 ingester を watcher 等で自動実行する** — AI 生成コードの無検証実行は RCE 級。マルチユーザ / 共有インスタンスでは特に不可。
- **宣言的マッピングを「レイアウト変更」として今すぐ導入する** — 後述の通り substrate の新規プロジェクトであり、軽い差し替えではない。

---

## 1. なぜ「宣言的マッピング」は今すぐには成立しないか

調査（実コード + [`linkml-vs-rdf-config.md`](linkml-vs-rdf-config.md)）で確認した事実:

- **rdf-config / LinkML は「スキーマ」を生成するツールで、CSV→RDF の ETL エンジンではない。** 出すのは ShEx / OWL TBox / Mermaid / JSON-Schema、せいぜい Python クラスの scaffold まで。実際の行→トリプル変換は生成しない（linkml-vs-rdf-config.md §3.4「Python class は要再評価」「ingester の主たる責務は CSV row を triple に変換すること」）。
- **実際の変換は手続き的な ingester が担う。** starrydata の正規 ingester は約 800 行で、dataset 固有の処理の塊:
  - セル内 JSON のパース（著者配列 / sample_info オブジェクト / x・y 数値配列）
  - 独自フォーマットの日付パース（タイムゾーン名除去を含む）
  - QUDT の「文字列 → IRI」ルックアップ
  - IRI 不正文字のパーセントエンコード・絶対 IRI でない値の triple スキップ
  - 複合キーの IRI 生成・bnode 排除
- **この「難しい 8 割」は純粋な宣言的マッピング（RML の triple map 等）では表現できない。** RML で表現するには結局カスタム関数（FnO）＝コードが要る。それを dataset ごとに生成すれば RCE が戻る。

つまり「マッピング＝データ」にできるのは「どの列 → どの述語」の対応までで、変換ロジック本体は宣言化しきれない。

## 2. セキュリティ原則（最重要）

**AI が生成した ingester を自動実行しない。** 理由:

- 生成コードは無検証では信頼できない（誤変換・無限ループ・意図せぬ I/O・悪意ある CSV に誘導された有害コード）。
- 想定ユーザに「研究室内の他の研究者・共同研究者」が含まれ、共有インスタンス上で他者のアップロードに紐づく生成コードが走る構図は、実質マルチテナント RCE。
- ソブリン self-host でも、無検証実行はインシデント時の説明責任を失う。

→ 取り込み経路は常に、「人間レビュー + 人間が実行」を挟む（option 1）か、「codegen を経路から排除」する（option 2）かのどちらか。中間（自動実行）は採らない。

## 3. 3 択と判断

| 案 | 内容 | コスト | 担当 | 安全性 |
|---|---|---|---|---|
| **1. 人間承認ゲート（暫定採用）** | 生成 ingester を人間がレビュー → 自分で実行（既存 watcher 経由）。UI は DL → 受け渡し →「取り込み済みを Ask」を繋ぐだけ | 小 | UI で可 | 安全（人間が gate） |
| **2. 宣言的マッピング + 検証済みエンジン（北極星・Phase 5）** | 変換を表現できる宣言的マッピング言語 + 実行エンジン + **閉じた検証済み関数ライブラリ**（新しい変換は人間が一度だけ追加）。データ経路から codegen を消す | 大 | core 主導 | 安全（コード実行なし）だが構築が重い |
| 3. 妥協なし即時 | 宣言エンジンは存在しないので不可 | — | — | — |

**判断: 1 を暫定の既定とし、2 を北極星として Phase 5 に置く。**

option 2 を成立させるには最低でも、(a) 変換を表現できる十分にリッチな宣言的マッピング言語、(b) それを実行する検証済みエンジン、(c) 変換関数を固定の検証済みライブラリとして閉じる（RML+FnO の関数集合を閉じ、新規変換が要る dataset は人間が一度だけライブラリに足す）、が要る。これは取り込み基盤（ingest）の設計＝core の領域であり、UI が単独で着手すべきものではない。

## 4. なぜ Phase 5（2 個目 dataset）まで後ろ倒すか

- option 2 の肝は「閉じた関数ライブラリ」だが、その範囲は**実際に複数 dataset で再発する変換**から決めるべき。starrydata だけ（n=1）で設計すると、ライブラリが過不足になる。
- Supercon（2 個目）を実投入して初めて、どの変換が共通でライブラリ化に値するかが見える。
- よって設計・実装は「2 個目 dataset 投入」とセットの Phase 5 とする。

## 5. デモ（ARiSE）との関係

- デモは投入済みの starrydata を Ask / 来歴で引くだけ。**取り込みループ（option 1/2）は demo の critical path に無い。**
- step0 の「CSV → AI がスキーマ提案 → 人間レビュー」（M1 ワークベンチ）は設計の協働であり、取り込み（実行）を伴わない。デモの見せ場はここまでで成立する。
- 取り込みループを demo に入れたい場合のみ、option 1（人間ゲート）を UI 側で繋ぐ。自動実行はしない。

## 6. 担当境界

- **取り込み基盤（宣言エンジン・関数ライブラリ）= core の領域。** UI は取り込みエンジンを作らない。
- **UI の安全な役割 = option 1 の配線**（設計 → 生成物 DL → 人間が watcher で取り込み → 履歴 → Ask）。

## 7. 残課題（Phase 5 で解く）

- [ ] 宣言的マッピング言語の選定（RML / RML+FnO / 独自）と、starrydata の変換がどこまで宣言化できるかの実測。
- [ ] 検証済み関数ライブラリの初期集合（date パース / JSON 展開 / QUDT ルックアップ / 複合 IRI / IRI sanitize）を Supercon と突き合わせて確定。
- [ ] 人間ゲート（option 1）の運用 UX（生成 ingester の diff レビュー、承認ログ）。
- [ ] 「閉じた関数集合」を破る dataset が来たときの追加フロー（人間が 1 回ライブラリに足す）の手順化。
