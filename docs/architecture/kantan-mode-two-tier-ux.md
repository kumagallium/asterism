# かんたん/詳細 二層モード — データセット追加 UX の再設計

status: 決定（ユーザー承認済 2026-07-21）→ 実装は段階的（先行条件 → UI 骨組み → 自動連結 → 試し検索）
関連: [`phase5-workbench-materialize-gate.md`](phase5-workbench-materialize-gate.md)（本 ADR が D1 を改訂）/ [`mapping-ir-phase2b-skeleton-wizard.md`](mapping-ir-phase2b-skeleton-wizard.md)（骨格ゲートを流用）/ [`ask-quality-and-generality.md`](ask-quality-and-generality.md)

## 0. 問題（開発者本人が「作った自分でも分からない」）

データセット追加フローが情報過多・専門用語過多で、オントロジーを知らない・IT リテラシーの低い実験科学者が初見で完走できない。12 エージェント分析（5 読解 → 3 設計案 → 3 視点審査 → 統合）による診断は 3 点:

1. **承認儀式**: 機械が既に守っているもの（422 ハードゲート `assert_rml_safe`/`validate_rml_design`・T1–T9 自動修正・draft 隔離・rollback）まで人間の承認ステップとして画面に並べており、承認文が「RML / Morph-KGC / Tier 0 / canonical」という実装名の弁明で埋まる（`workbench.json` `ingest.note` は 1 文に専門概念 6 語）。説明文は各機構が main に入るたびに「存在証明」として追記されたもので、「あなたは今どこにいて、あと何をすれば完了か」を語らない。
2. **進捗モデル不在**: 3 段ステッパーが「保存」で終点。取り込み・公開はカタログ側の別画面に分断され、promote 完了から Ask への直リンクも無い。実測: 初回公開まで能動的判断 20 回（Workbench）+12 回（カタログ）。
3. **入口の門前払い**: 最初の関門が LLM API キー入力（sk-… の概念ごと）。実験科学者の主データ形式 .xlsx を受理しない（`datasetsApi.ts` `TABULAR_ACCEPT`）。

## 1. 決定事項

| # | 論点 | 決定 | 理由 |
|---|---|---|---|
| K1 | 情報設計 | **二層モード**: 既定=「かんたんモード」（5 工程レシピカード常駐・9 画面）／「詳細モード」=現行 Workbench/カタログ全機能を**削除ゼロ**で温存。トグルで同一 dataset/job のまま往復 | 審査 3 視点（ペルソナ本人・品質の番人・UX リサーチャー）全員一致。初見者の完走と上級者の全能力を両立 |
| K2 | 人間ゲートの集約 | 人間の確認は **3 問だけ**: ① 行の数えかた（=骨格ゲート）② 列の意味（=設計レビュー）③ 公開（=promote）。いずれも「データの意味・公開意思」という**人間しか知り得ない事柄**を守る | 機械が保証できるもの（実行安全=422、引用面の純度=draft 隔離+promoted フラグ）を人間の承認から外し、残った 3 つだけを平易な質問形で前面化 |
| K3 | 保存→draft 取り込みの自動連結 | かんたん層では materialize 成功後に **draft への ingest を自動実行**（「確定して保存」「データを取り込む（承認）」の 2 ボタンを 1 工程に畳む）。**[`phase5-workbench-materialize-gate.md`](phase5-workbench-materialize-gate.md) D1（2 段ゲート）を改訂** | D1 制定時（#15）に承認ボタンが守っていたリスクは、その後の 422 ハードゲート（実行前検証）+ draft 隔離 + rollback finally 化 + 孤児 version graph 回収で**構造的に消えた**。未 promote データは誰にも見えず Ask も引用しない。詳細モードの明示ボタンは温存 |
| K4 | 用語ポリシー | かんたん層に fatal 級専門用語を**出さない**（RML・IRI・名前空間・canonical・draft graph・述語・クラス・トリプル・Morph-KGC・Tier 0…）。「グラフ」の語も使わない。同一概念の複数呼称（述語=プロパティ、設計図=語彙=スキーマ、共有データ=canonical=正式グラフ）は**層ごとに 1 本化**。**適用範囲の拡大（2026-07-23 決定）**: かんたんウィザードだけでなく**共有画面（ホーム/データセット/つながり/共通の言葉/引用カード等）にも同じ対訳を適用**する — かんたんで作った人が公開直後にカタログで旧用語に出会う断絶が実測されたため。専門表記の逃がし先は詳細ビューア（ルールタブの原文（RML 形式）・型列・高度な検索・詳細モード）に限定 | 「説明を読んでも分からない」の根因=説明の主語が信頼モデル（内部構造の弁明）。かんたん層の説明は常に「ユーザーの目的」を主語にする。共有画面は初見者も踏むため、かんたん側に 1 本化する |
| K5 | LLM 設定の撤去 | かんたん層から キー/モデル/base URL の概念を**全廃**。サーバ側キー（`/api/llm/server-keys`・実装済）を既定とし、未設定サーバのみ「管理者が AI の設定を済ませると使えます」1 枚（個人利用者にはファーストラン 1 画面） | 入口の門前払い解消。設定モーダルは詳細モードに温存 |
| K6 | .xlsx 受理 | .xlsx をサーバ側で表形式へ**決定論変換**して受理（変換ツール・版は来歴に記録）。多シート等で一意に決められない場合のみ「どのシートを使いますか？」の汎用質問にフォールバック | 実験科学者の主データ形式。docx/PDF→XML のサーバ側変換と同じ「vet 済み決定論アダプタ」パターン |
| K7 | 骨格ゲートの規律 | 全行一意でも**必ず 1 タップ確認**（auto-continue しない・[`mapping-ir-phase2b-skeleton-wizard.md`](mapping-ir-phase2b-skeleton-wizard.md) §9(b) 維持）。「重なったまま進む」を既定ボタンにしない。複数 map 時は緑の map を畳んでよいが、**畳んだ map にも「1 行=◯◯」の読み取り文を必ず表示**（意味誤り×偶然一意の素通り防止） | 品質の番人審査の最重要指摘。骨格ゲートは弱モデルが主キーを誤る実測があり（Phase 2b dogfood）、形骸化させない |
| K8 | 列の意味の出所 | Mapping IR の property 行に **optional `label`（人間可読の意味）/ `unit`（単位）** を追加し、propose に書かせる。S6「列の意味」テーブルと ontology projection の rdfs:label 補強がこれを消費。**散文（§1–8）からの投影代用は不可** | 現 IR に意味ラベルの置き場が無い（`mapping_ir_schema.py` 実測）。決定論投影の原則（AI を再度通さない）を守る |
| K9 | ためす（試し検索） | draft 取り込み直後に**決定論クエリ（件数・範囲・最大値+出典 IRI）を自動実行して見せる**。任意ボタンにしない（初見者は押さない）。生成できないデータ形は種類別件数+実 ID にフォールバック | 「引用できる事実」の価値をユーザー自身のデータで最初に体感させる。列の意味の誤りをここでも発見できる |
| K10 | 公開経路の一本化 | 公開は確認ダイアログ（名前・種類別件数・ことばの要約・撤回可能性の約束）**一本**。未確定項目が 1 つでも残る限りボタン無効。カタログカード側のワンクリック公開は**初回公開では全モードでこのダイアログ必須** | 全自動案の審査で出た公開バイパスの穴を先に塞ぐ |
| K11 | 非収束時の出口 | 自動修正で解決できない場合のみ停止カード 1 枚。**全エラー系統→平易質問の翻訳テーブルを実装前に仕様化**（例: T2→「Excel で保存した CSV の文字化け対策をしています」）。出口は常に 2 つ: [AI にもう一度直させる][詳細モードで自分で直す] | 初見者の行き止まりを作らない。T1–T9 の生ラベルが「進行不能の主因」という用語棚卸しの実測に対応 |
| K12 | 件数の表現 | かんたん層はトリプル数を出さず**種類別件数**（「試料 24 件・測定 500 件」）。S6 に「元ファイル N 行 → …」の対応カードを常時表示 | 主キー潰れをユーザーが自分の目で発見できる（行数↔件数対応が最強の検証面） |
| K13 | 命名の決定論化（2026-07-23 追加） | mint 名前空間で**人間が判断する名前は「データセットの名前」（IRI 内 slug）1 つだけ**。CURIE prefix（略記）は slug から**機械導出**（`derive_prefix_pair`: 先頭トークン→衝突時トークン連結→`ds` 保険・resource は +`r`・NCName 安全）し、AI にも人間にも選ばせない。AI 提案骨格はサーバで**正規形に正規化**（`normalize_dataset_namespace`: `{base}/datasets/{slug}/ontology#`+`resource/` へ base/形ずれ・placeholder mint を修復、CURIE 連動 rename）。骨格ゲートは「データセットの名前」1 欄（rename cascade・編集は他のキー編集同様に再検査）+「ID の発行元」（base 未設定なら仮発行元警告+設定への案内。**base は設定の領分 — ゲートで編集させない**）。かんたん層は生 prefix 表を出さず（K4）、詳細層は「高度」details に温存 | prefix は保存データに一切現れない純表記で、「何をつけたら良いか全くわからない」は正当な反応（2026-07-23 ZEM dogfood 実測）。意味を持つ判断（slug=将来この箱に何を足すか）だけを人間に残し、意味を持たない判断は機械が引き受ける — 修正レシピ（T1-T9）と同じ「機械的要件は機械が用意する」原則 |

## 2. かんたんモードの画面フロー（9 画面）

常駐レシピカード「**① 入れる → ② AI が読む → ③ 確かめる → ④ ためす → ⑤ 公開する**」（現在地点灯。draft 取り込みは工程名に出さない=自動）。

| 画面 | 内容 | 裏で走る工程 | ゲート |
|---|---|---|---|
| S1 はじめる | ドロップ欄（Excel/CSV/装置ファイル/Word/PDF）。「公開するまで誰にも見えません」。サンプルデータ導線 | 種別自動判定・.xlsx 変換・サーバキー確認 | なし |
| S2 読み取りの確認 | 先頭 5 行プレビュー+「あなたしか知らない 2 問」（メタ行の要否・ID の通用範囲。証拠つき・該当時のみ） | dialect 自動検出。回答を preamble/複合キーに注入 | 人間知識の前借り |
| S3 AI が読んでいます | 自分の表を常時表示+平易実況。「閉じても続きます」 | `POST /api/propose/skeleton`（生成方式二択は廃止） | なし |
| S4 行の数えかた | 「1 行=1 つの試料の 1 回の測定」+一意性の実データ証拠（緑帯/赤帯+候補チップ） | skeleton/validate 再検査 | **本丸①**（K7） |
| S5 自動処理 | 実況（列の意味→品質チェック 8 項目→下書き取り込み N/M 行） | per-map continue→T1–T9 自動修正→materialize→source 永続化→**draft 自動 ingest**（K3） | 非収束時のみ停止カード（K11） |
| S6 列の意味 | 対応カード（K12）+列テーブル（列名/意味/単位/実例/判定）。低確信は黄マーカー=視線誘導のみ。全列能動確定 | IR label/unit 消費（K8）。修正は refine へ構造化コメント | **本丸②** |
| S7 ためす | 自動試し検索 3 問+出典リンク（K9） | draft スコープ決定論クエリ | 実質ゲート（戻る導線） |
| S8 公開する | 名前・種類別件数・ことば要約・撤回可能性。未確定あればボタン無効 | promote+alignment を 1 画面統合（K10） | **本丸③** |
| S9 できあがり | 試し質問チップ（Ask へプリフィル遷移）+「新しい測定分を足す/全部を差し替える」 | append 自動判定・reingest+repromote の 1 ウィザード化 | なし |

## 3. 実装順（PR 分割）

1. **先行条件（独立・並行可）**: (a) IR label/unit（K8） (b) .xlsx 変換（K6） (c) 本 ADR+ROADMAP。
2. **UI 骨組み**: モード切替+レシピカード+S1–S4（骨格ゲート機構は既存流用）。LlmGate はサーバキー有時に非表示（K5）。
3. **自動連結**: K3（保存→draft ingest）+S5+S6。同 PR で `phase5-workbench-materialize-gate.md` D1 改訂を確定。
4. **S7–S9**: 試し検索 API+公開統合+完了画面。
5. **翻訳テーブル**: K11 の仕様化+配線。

## 4. 変えないもの（不変条件の確認）

- **決定論・引用可能性**: どの事実も出典 IRI に遡れる。Ask は typed ツール（LLM-free）のまま。
- **実行安全**: 生成コードは走らない。Morph-KGC+Tier 0 閉集合+422 ハードゲートは完全無傷（不可視化するだけ）。
- **汎用性**: 材料科学特化の仕掛けは入れない（画面例文はプレースホルダ。ロジックは全ソース共通）。
- **詳細モード**: 現行機能の削除ゼロ。生 RML/Turtle・T1–T9 格子・SPARQL・crosswalk・個人キー設定は全て詳細層に残る。

## 5.1 K11 翻訳テーブル（停止カードの正準・fail 時の 1 行）

S5 停止カード（`kind: design`）の**表示はこの平易文**、**AI 修正へ渡す本文は従来の技術文+修正レシピのまま**（表示と AI 入力の分離 — 平易文だけでは弱モデルが着地できない、という修正レシピ機構の教訓を守る）。実装は `locales/{ja,en}/kantan.json` の `s5.trap.*`（この表が正準・両言語同時反映）。

| 系統 | かんたん層の 1 行（fail の意味） |
|---|---|
| T1 ID uniqueness | 行ごとの ID が重なってしまいます（重なった行は 1 件にまとまります） |
| T2 BOM | Excel で保存した CSV で文字化けするおそれがあります |
| T3 bnode-free | 名前（ID）の付いていないデータが残っています |
| T4 MIE keywords / categories | 検索タグ（このデータを探すときの言葉）が足りません |
| T5 Mermaid syntax | 設計の見取り図がうまく描けていません |
| T6 fake sample_rdf_entries | 設計に添える実例が、実際の設計と食い違っています |
| T7 Why / Alternatives / Trade-offs | 設計の理由の説明が欠けています |
| T8 AI hallucination | AI が、ファイルにない列を使おうとしました |
| T9 RML closed-set | 変換ルールに、安全確認済みでない操作が混ざっています |
| incomplete（truncation） | AI の設計の出力が途中で切れています |
| その他（warnings / validation・mapping issues） | 個別翻訳しない（自由文のため）。「このほか、取り込みルールへの変換で◯件の問題があります」と**件数でまとめる**。全文は「詳しい内容（技術情報）」に温存 |
| 未知の trap id | 将来 T10+ が増えた場合は「その他」件数に合流（fail-open な文言事故を作らない） |
